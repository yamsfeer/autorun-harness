## 高优先级 (Phase 1)

### 1. 文档索引系统 (CLAUDE.md)
- **来源**：用户需求
- **描述**：建立项目文档索引，作为 Agent 的持续上下文
- **核心内容**：
  - PRD 文档摘要（完整 PRD 太长，提供摘要）
  - **文档索引**：
    - 完整 PRD 文档位置
    - 数据契约文档（前后端 API）
    - 数据实体（数据库表结构）
    - UE 交互逻辑状态机
    - 业务流程图 flowchart
    - 设计系统 DESIGN.md
- **相关文件**：`CLAUDE.md`
- **时机**：项目初始化时由 Planner 生成

### 2. 主控制脚本
- **描述**：编排整个流程的主控程序
- **用途**：循环启动 Agent SDK、管理会话、更新状态
- **相关文件**：`src/orchestrator.ts` 或 `run.sh`
- **功能**：
  - 检测任务状态
  - 调用对应 Agent
  - 处理会话重置
  - 记录日志

### 3. init.sh 脚本模板
- **描述**：创建标准化的项目初始化脚本模板
- **用途**：启动开发服务器、安装依赖、数据库迁移等
- **相关文件**：`templates/init.sh`
- **备注**：需要支持不同技术栈（React/Vue/Node 等）

### 4. 会话启动提示词
- **描述**：每轮任务开始时的状态恢复指令
- **用途**：让新会话快速了解项目状态并开始工作
- **相关文件**：`prompts/session_start.md`
- **内容要点**：
  - 读取 CLAUDE.md 获取项目概览
  - 读取 tasks.json 获取任务状态
  - 读取 progress.txt 获取历史进展
  - 如何选择下一个任务
  - 如何处理依赖检查

## 中优先级 (Phase 2)

### 5. 错误处理机制 ✅
- **描述**：处理各种异常情况的机制
- **实现状态**：已完成
- **场景**：
  - 网络错误、API 超时
  - Agent SDK 调用失败
  - 开发服务器启动失败
  - 测试环境问题
- **相关文件**：`src/core/error-handler.ts`
- **功能**：
  - 错误类型分类（rate_limit、usage_limit、network、api_timeout）
  - 重试机制（withRetry）
  - 超时控制（withTimeout）
  - 提供商切换判断（shouldSwitchProvider）

### 6. failure.md 错误收集系统 ✅
- **来源**：用户需求
- **描述**：收集开发过程中的错误和失败情况
- **实现状态**：已完成
- **用途**：
  - 添加到 CLAUDE.md，保证后期不再犯同样错误
  - 生成收集报告供分析
- **相关文件**：`src/core/failure-collector.ts`
- **收集内容**：
  - 关联具体任务和验收标准
  - 失败原因和错误类型
  - 修复尝试记录
  - 生成 .harness/failure.md

### 7. 目录结构规范 ✅
- **描述**：项目的标准目录布局
- **用途**：确保所有 Agent 使用一致的文件位置
- **实现状态**：已完成
- **实现结构**：
  ```
  project/
  ├── CLAUDE.md              # 文档索引（Claude Code 入口）
  ├── docs/                  # 完整文档目录
  │   ├── PRD.md             # 原始 PRD
  │   ├── DESIGN.md          # 设计系统
  │   ├── API_CONTRACT.md    # 前后端 API 契约
  │   ├── DATA_MODEL.md      # 数据实体（数据库表结构）
  │   ├── UE_FLOW.md         # UE 交互逻辑状态机
  │   └── FLOWCHART.md       # 业务流程图
  ├── .harness/
  │   ├── spec.md            # 技术规格（简洁版）
  │   ├── tasks.json         # 任务列表
  │   ├── progress.txt       # 进度日志
  │   ├── reports/           # 评估报告
  │   └── screenshots/       # 测试截图
  ├── init.sh                # 初始化脚本
  └── README.md              # 项目说明
  ```
- **相关文件**：
  - `src/commands/init.ts` — 创建目录结构
  - `prompts/planner.md` — 生成文档结构

### 8. 日志系统 ✅
- **描述**：统一的日志记录和查看系统
- **实现状态**：已完成
- **用途**：追踪每次会话的执行情况
- **相关文件**：`src/core/logger.ts`
- **功能**：
  - 结构化日志（JSON 格式）
  - 支持控制台和文件输出
  - 日志级别：debug/info/warn/error
  - 时间戳、耗时统计

### 9. 成本监控 ✅
- **描述**：Token 使用量追踪
- **实现状态**：已完成
- **用途**：防止成本失控
- **相关文件**：`src/core/cost-tracker.ts`
- **功能**：
  - 记录每次 API 调用的 token 数
  - 按 Agent 和任务分组统计
  - Token 预算控制
  - 生成成本摘要报告

### 10. 多服务提供商管理 ✅
- **描述**：支持多个 AI 服务提供商的池化管理
- **实现状态**：已完成
- **用途**：
  - 遇到 429 错误时自动切换提供商
  - 负载均衡
  - 避免单点故障
- **相关文件**：`src/core/provider-manager.ts`
- **配置位置**：`~/.config/autorun-harness/providers/*.json`
- **功能**：
  - 全局提供商配置管理
  - 自动切换逻辑
  - CLI 命令：`autorun-harness provider --add/--list/--switch`

### 11. 人工介入接口
- **描述**：当任务标记为 needs_human 时的处理机制
- **用途**：让人类开发者介入修复
- **相关文件**：`src/human_intervention.ts`
- **功能**：
  - 通知机制（邮件/Webhook）
  - 人工修复入口
  - 恢复自动执行

## 中低优先级 (Phase 3)

### 12. Skills 库
- **来源**：用户需求
- **描述**：提供技能库，由 Agent 判断是否要取用
- **用途**：
  - 按需安装相关 skill，而非一次性全部安装
  - 例如：支付相关 skill，当 task 是开发支付模块时才安装
- **相关文件**：`skills/`
- **设计要点**：
  - Skills 元数据描述
  - 任务与 Skill 的匹配逻辑
  - Skill 安装/卸载机制
- **示例 Skills**：
  - `payment` - 支付集成
  - `auth` - 认证授权
  - `analytics` - 数据分析
  - `notification` - 通知推送

### 13. Agent Browser 替代 Playwright
- **来源**：用户需求
- **描述**：用 Agent Browser 替代 Playwright 做测试
- **优势**：
  - 更自然的测试交互
  - 无需预定义选择器
  - 更灵活的错误处理
- **相关文件**：评估器提示词更新
- **待定**：
  - Agent Browser 的具体实现方式
  - 与现有评估流程的集成

### 14. 多账号切换
- **来源**：用户需求
- **描述**：支持在多个 CodeBuddy 账号间切换
- **用途**：
  - 负载均衡
  - 不同项目的账号隔离
  - 绕过单个账号的速率限制
- **相关文件**：`src/account_manager.ts`
- **功能**：
  - 账号配置管理
  - 自动切换逻辑
  - 使用统计

### 15. 进度可视化
- **来源**：用户需求（可选）
- **描述**：tasks.json 和 progress.txt 的直观展现
- **用途**：方便用户了解项目状态
- **相关文件**：`dashboard/`
- **功能**：
  - Linear 看板风格
  - 任务列表和状态
  - 进度统计图表
  - 实时日志流
  - 手动干预入口
