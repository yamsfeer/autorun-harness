# 长期运行代理框架设计

## 背景

本设计融合了 Anthropic 两篇工程博客的核心思想：
- 《Effective harnesses for long-running agents》——任务拆分 + 循环重置
- 《Harness design for long-running application development》——多代理协作 + 对抗式评估

目标：让 AI 代理能够在无需人工干预的情况下，完成需要数小时甚至数天工作的大型软件开发项目。

---

## 一、核心问题分析

### 1.1 长期运行代理的挑战

| 问题 | 表现 | 根因 |
|------|------|------|
| 上下文窗口限制 | 复杂项目无法在单次会话完成 | 模型有 token 上限 |
| 上下文焦虑 | 接近上下文限制时过早收尾 | 模型担心空间不够 |
| 记忆丢失 | 新会话不知道之前做了什么 | 会话间无状态传递 |
| 任务膨胀 | 试图一次性做太多事情 | 缺乏约束和分解 |
| 过早宣布完成 | 未真正完成就声称做完 | 缺乏验收标准 |
| 自我评估不可靠 | 自评时过于宽容 | "自己检查自己"有盲区 |

### 1.2 两篇文章的解决方案对比

| 维度 | 第一篇 | 第二篇 |
|------|--------|--------|
| 代理数量 | 2个角色（初始化器 + 编码器轮班） | 3个独立代理（规划器、生成器、评估器） |
| 协作模式 | 顺序接力，文件传递 | 实时对抗，反馈迭代 |
| 上下文管理 | 每次任务后重置 | 连续运行 + 压缩（依赖模型能力） |
| 核心创新 | 结构化交接产物 | 独立评估器监督 |
| 成本 | 相对较低 | 高（约20倍于单代理） |
| 适用模型 | Sonnet 4.5 等中等能力模型 | Opus 4.5/4.6 等高能力模型 |

### 1.3 我们的取舍

结合当前模型能力和成本考虑：

| 选择 | 理由 |
|------|------|
| 采用循环重置 | 避免"上下文焦虑"，每次会话都是"新鲜"状态 |
| 引入独立评估器 | 解决自我评估不可靠问题 |
| 规划器前置生成详细规格 | 降低用户输入门槛，只需粗略PRD |
| 任务内连续交互，任务间重置 | 平衡效率与上下文管理 |

### 2.2 代理角色定义

| 代理 | 职责 | 输入 | 输出 | 运行时机 |
|------|------|------|------|----------|
| **规划器** | 理解需求，拆分任务 | PRD文档 | spec.md, tasks.json | 项目开始时一次 |
| **生成器** | 实现功能代码 | 任务详情 + spec.md + 评估反馈 | 代码变更 | 每个任务执行时 |
| **评估器** | 验收测试，评分反馈 | 任务详情 + 代码状态 | evaluator_report.json | 每次生成器提交后 |

### 2.3 为什么这样设计？

#### 为什么需要规划器？

| 考量 | 说明 |
|------|------|
| 降低用户门槛 | 用户只需提供粗略PRD，不需要写出精确的200条功能 |
| 知识注入 | 规划器可以结合领域知识，补充用户未想到的边界情况 |
| 一致性保证 | 所有后续代理都基于同一份规格文档工作，避免理解偏差 |
| 前置思考 | 强迫在编码前思考清楚"要做什么"，避免边做边改 |

#### 为什么需要独立评估器？

```
自我评估的问题：
┌─────────────────────────────────────┐
│  生成器：我实现了登录功能            │
│  生成器自我评估：看起来不错，通过！   │
│  实际情况：错误密码不报错、空输入崩溃 │
└─────────────────────────────────────┘

独立评估器的优势：
┌─────────────────────────────────────┐
│  生成器：我实现了登录功能            │
│  评估器：让我测一下...               │
│    - 错误密码 → 没有提示 ❌          │
│    - 空输入 → 页面崩溃 ❌            │
│    - 正确密码 → 登录成功 ✓           │
│  评估结果：不通过，需要修复以下问题   │
└─────────────────────────────────────┘
```

**核心理念**：将"做工作"和"评判工作"分离，类似GAN中生成器和判别器的对抗关系。

#### 为什么选择循环重置而非连续运行？

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 连续运行 | 流程连贯，无交接开销 | 上下文累积，可能偏离 | 高能力模型 + 短任务 |
| 循环重置 | 每次状态干净，不易跑偏 | 有交接开销，需要结构化产物 | 中等能力模型 + 长任务 |

**选择理由**：
1. 当前模型仍存在"上下文焦虑"问题
2. 大型项目任务多，上下文累积风险高
3. 结构化产物（tasks.json、progress.txt）本身就是好实践
4. 重置后的"冷启动"成本可通过良好的交接产物降到最低

### 3.2 spec.md — 产品规格文档

规划器生成，作为所有代理的"设计蓝图"。

```markdown
# 用户管理系统 — 产品规格

## 1. 项目概述

### 1.1 目标
构建一个完整的用户管理系统，支持用户注册、登录、个人信息管理等功能。

### 1.2 技术栈
- 前端：React 18 + TypeScript + Tailwind CSS
- 后端：Node.js + Express + TypeScript
- 数据库：PostgreSQL
- 认证：JWT

### 1.3 约束条件
- 响应式设计，支持移动端
- API 响应时间 < 200ms
- 支持 1000 并发用户

## 2. 功能模块

### 2.1 认证模块

#### 2.1.1 登录功能
- 页面路由：/login
- 输入字段：邮箱、密码
- 交互行为：
  - 空字段时禁用登录按钮
  - 错误凭据显示错误提示
  - 成功登录跳转到仪表盘
- API 端点：POST /api/auth/login
- 错误处理：
  - 400：参数格式错误
  - 401：认证失败
  - 500：服务器错误

#### 2.1.2 注册功能
...

## 3. 数据模型

### User
```typescript
interface User {
  id: string;
  email: string;
  password_hash: string;
  created_at: Date;
  updated_at: Date;
}
```

## 4. API 设计

| 端点 | 方法 | 描述 | 认证 |
|------|------|------|------|
| /api/auth/login | POST | 用户登录 | 否 |
| /api/auth/register | POST | 用户注册 | 否 |
| /api/users/me | GET | 获取当前用户 | 是 |

## 5. UI 设计规范

### 5.1 颜色
- 主色：#3B82F6 (蓝色)
- 成功：#10B981 (绿色)
- 错误：#EF4444 (红色)
- 警告：#F59E0B (黄色)

### 5.2 字体
- 正文：Inter, 16px
- 标题：Inter, 24px, bold

...
```

**spec.md 的作用**：

| 作用 | 说明 |
|------|------|
| 设计约束 | 所有代理都遵循同一份设计，避免"各做各的" |
| 决策记录 | 技术选型、API设计等决策在此统一管理 |
| 参考手册 | 生成器实现时查阅，评估器验收时对照 |
| 减少猜测 | 具体的规格比模糊的需求更易执行 |

## 四、执行流程详解

### 4.1 初始化阶段

```
输入：用户提供的 PRD 文档（可以是粗略的）
输出：spec.md, tasks.json, init.sh, progress.txt, git 仓库

步骤：
1. 读取 PRD 文档
2. 规划器分析需求，补充遗漏的功能点
3. 生成详细的产品规格文档 spec.md
4. 将功能拆分为任务，生成 tasks.json
5. 创建项目初始化脚本 init.sh
6. 创建空的 progress.txt
7. 初始化 git 仓库，做初始提交
```

**规划器的关键职责**：

| 职责 | 说明 |
|------|------|
| 需求理解 | 从粗略PRD中提取核心需求 |
| 需求补充 | 补充用户未想到的边界情况、错误处理 |
| 功能拆分 | 将大功能分解为可执行的小任务 |
| 验收定义 | 为每个任务定义清晰的验收标准 |
| 依赖分析 | 确定任务之间的依赖关系 |
| 技术设计 | 确定技术栈、API设计、数据模型 |

### 4.3 关键决策点

#### 决策 1：何时重置上下文？

| 时机 | 选择 | 理由 |
|------|------|------|
| 任务完成后 | **重置** | 任务之间相对独立，新任务新上下文避免累积问题 |
| 生成器→评估器 | **不重置** | 评估器需要理解生成器的实现，保持上下文连贯 |
| 修复迭代 | **不重置** | 修复需要理解之前的评估反馈，连续执行效率更高 |

#### 决策 2：如何处理多次失败？

```
attempts 计数逻辑：
- 每次评估器返回 fail，attempts++
- attempts < 3：允许继续修复
- attempts >= 3：标记 needs_human，跳过

为什么是 3 次？
- 1 次：可能是偶发问题，值得重试
- 2 次：给了修复机会，可能方向有误
- 3 次：问题可能超出当前能力，强求会浪费时间
```

#### 决策 3：任务依赖如何处理？

```json
// 示例：T002 依赖 T001
{
  "id": "T002",
  "dependencies": ["T001"]
}
```

```
任务选择逻辑：
1. 找到所有 status="pending" 的任务
2. 过滤：dependencies 中的任务都已 completed
3. 按 priority 排序（high > medium > low）
4. 选择第一个符合条件的任务
```

## 五、评估器评分体系

### 5.1 评分维度

| 维度 | 权重 | 评估内容 | 评分要点 |
|------|------|----------|----------|
| **functionality** | 40% | 功能是否正常 | 验收标准通过率 |
| **code_quality** | 25% | 代码质量 | 可读性、错误处理、最佳实践 |
| **product_depth** | 20% | 产品完整性 | 边界情况、用户体验细节 |
| **visual_design** | 15% | 视觉设计 | 符合设计规范、布局合理性 |

### 5.2 评分规则

```
单项评分：
- 1.0 = 完美，无问题
- 0.8 = 良好，有轻微改进空间
- 0.6 = 及格，有明显问题但基本可用
- 0.4 = 较差，存在较大问题
- 0.2 = 很差，基本不可用
- 0.0 = 完全失败

加权总分：
total_score = Σ(score_i × weight_i)

通过标准：
- total_score >= 0.75：通过
- 0.5 <= total_score < 0.75：需要修复
- total_score < 0.5：可能需要重做

特殊规则：
- functionality 维度任一验收标准 fail → 整体 fail
- 存在崩溃、安全漏洞 → 直接 fail
```

### 5.3 评估器提示词核心原则

```
1. 严格而非宽容
   - 宁可多挑刺，不要放过问题
   - "差不多"不等于"通过"

2. 具体而非笼统
   - 指出具体文件、行号、错误信息
   - 反馈要足够具体，开发者能直接修复

3. 完整而非片面
   - 必须执行每个 steps
   - 不要因为"看起来没问题"就跳过

4. 实证而非臆断
   - 必须实际运行测试，不能只看代码
   - 检查控制台错误、网络请求

禁止行为：
- 不要因为"看起来还行"就通过
- 不要跳过任何验收步骤
- 不要忽略控制台错误或警告
- 不要自行降低标准

必须行为：
- 必须使用 Playwright 实际操作
- 必须检查控制台是否有错误
- 必须验证边界情况（空输入、特殊字符等）
- 必须截图记录问题
```

## 六、风险与缓解措施

### 6.1 潜在风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 规划器生成的任务粒度不当 | 中 | 任务太大做不完，太小开销大 | 迭代调优提示词，建立任务粒度标准 |
| 评估器过于宽松 | 高 | 问题遗漏到后续 | 强调严格原则，用少样本校准 |
| 评估器过于严格 | 中 | 无意义的反复修复 | 设置合理的通过阈值 |
| 任务间依赖复杂 | 中 | 执行顺序混乱 | 规划时明确依赖，运行时验证 |
| 成本过高 | 高 | 大型项目成本失控 | 设置预算上限，关键任务优先 |
| 规格文档与实现不一致 | 中 | 评估器按错误标准验收 | 保持 spec.md 版本控制，重大变更需更新 |

### 6.2 成本估算

```
假设项目参数：
- 任务数：20 个
- 平均每任务迭代：1.5 次（考虑修复）
- 每次生成器：约 100K input + 10K output tokens
- 每次评估器：约 50K input + 5K output tokens

单任务成本（Claude Opus 4.5 定价）：
- 生成器：(100K × $15 + 10K × $75) / 1M = $2.25
- 评估器：(50K × $15 + 5K × $75) / 1M = $1.13
- 单任务平均：$2.25 + $1.13 = $3.38
- 含修复(×1.5)：$5.07

总成本估算：
- 20 任务 × $5.07 = $101.4

对比：
- 单代理无框架：约 $5-10（但质量无法保证）
- 完整框架：约 $100+（质量可控）
```

### 6.3 效率优化建议

| 优化点 | 方法 | 预期收益 |
|--------|------|----------|
| 减少无效重试 | 评估器提供更精确的修复建议 | 减少 attempts 次数 |
| 批量评估 | 简单任务批量处理，减少会话数 | 降低会话启动开销 |
| 渐进式规格 | spec.md 按模块生成，非一次性全写 | 加快启动速度 |
| 智能跳过 | 已知技术难点提前标记 needs_human | 避免无效尝试 |

## 七、与原始设计的差异总结

| 方面 | 第一篇 | 第二篇 | 本设计 |
|------|--------|--------|--------|
| 初始化代理 | 有，设置环境 | 合并到规划器 | **规划器**，同时生成规格 |
| 编码代理 | 循环启动，文件交接 | 连续运行的生成器 | **生成器**，任务内连续 |
| 评估代理 | 无（自测） | 独立评估器 | **评估器**，任务内连续 |
| 上下文管理 | 每次重置 | 连续+压缩 | **任务间重置**，任务内连续 |
| 任务粒度 | 功能级 | 冲刺级 | **功能级**，更细粒度 |
| 成本控制 | 未讨论 | 高成本可接受 | **有意识控制**，设置阈值 |

## 附录：核心提示词模板

### A. 规划器提示词

```
你是产品规划专家，负责将用户需求转化为详细的开发规格。
