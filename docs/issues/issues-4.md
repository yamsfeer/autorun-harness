# 已知问题与解法

> 记录框架当前存在的结构性问题，以及对应的改进方向。
> 后续将通过评估体系量化验证每项改进的实际效果。

---

## 问题 1：上下文断裂

### 现状

每次调用 Agent SDK 是一个独立的会话。Generator 拿到的上下文只有：当前任务的 JSON 描述 + `spec.md`。但它要改的代码可能依赖之前任务写入的模块、接口、数据结构。

框架没有机制让 Generator 真正理解"当前代码库的全貌"——它只能靠 `Read` 工具临时去读文件，而读什么、读多少，完全取决于模型自己的判断。

**越往后的任务，上下文缺失越严重，出错概率指数级上升。**

### 影响范围

所有任务，尤其是有依赖关系的后半段任务。

### 对应改进

- 改进 1：依赖任务文件列表注入上下文
- 改进 4：维护活的架构摘要（ARCHITECTURE.md）
- 改进 5：Git diff 注入上下文
- 改进 7：两阶段上下文加载

## 问题 2：评估器可信度不足

### 现状

Evaluator 本身也是 AI，它给出 `pass` 不代表真的通过了。它会运行 `npm run build`，但不会做真正的边界测试。一个任务可以编译通过、基本功能看起来正常，但留下隐患（状态不一致、竞态条件、错误路径没处理）。

当这些隐患在后续任务中爆发时，很难追溯到根因。

**核心矛盾：AI 评估 AI，没有确定性检验手段。**

### 影响范围

所有需要评估的任务，直接影响任务通过率的可信度。

### 对应改进

- 改进 2：验收标准变成可执行测试用例（P0 优先级）

## 问题 3：错误积累

### 现状

框架有重试机制（最多 3 次），但重试是基于"同一任务同一上下文"的。如果第 3 个任务引入了一个微妙的设计缺陷，Evaluator 给了 pass，到第 8 个任务才发现问题，这时回退成本极高——框架没有跨任务的回归检测和回滚能力。

### 影响范围

任务数越多，积累效应越明显。中大型项目（>10 个任务）风险显著。

### 对应改进

- 改进 3：周期性全量回归测试
- 改进 6：Git tag 做回滚锚点

---

## 问题 4："最后 20%"瓶颈

### 现状

前 80% 的工作（项目骨架、CRUD、基本 UI）可以自动化。但最后 20%（边界情况、交互细节、性能调优、跨浏览器兼容）是占据 80% 时间的地方。

Evaluator 的评分体系（product_depth 20% + visual_design 15%）中这两个维度，AI 自我评估不可靠。

### 影响范围

复杂项目尤其明显。简单项目（Todo App 级别）基本不受影响。

### 对应改进

- 改进 2：可执行测试用例（至少能验证功能边界）
- 流程建议：分两阶段运行，Phase 1 自动化核心功能，Phase 2 人工介入打磨
