# 并行执行优化提案

> 创建日期：2026-04-16
> 状态：概念阶段
> 类型：架构改进 / 性能优化

## 背景

当前框架采用顺序执行模式：任务一个接一个执行，每个任务完成后才执行下一个。这种方式简单可靠，但未能充分利用任务间的并行潜力。

## 核心洞察

### 任务依赖图的本质

任务之间存在依赖关系，形成有向无环图（DAG）：

```
示例依赖结构：

A ─┬→ B ─┬→ E
   ├→ C ─┤
   └→ D ─┴→ F
```

**关键发现**：不依赖其他任务的任务（兄弟节点）之间是可以并行执行的。

---

## 优化方向一：任务级并行

### 目标

同一层级的任务并行执行，缩短总体执行时间。

```
传统顺序执行：
A → B → C → D → E → F
时间线：━━━━━━━━━━━━━━━━━━━━━━━━━━━

并行执行（兄弟节点并行）：
A ─┬→ B ─┬→ E
   ├→ C ─┤
   └→ D ─┴→ F
时间线：━━━━━━━━━━━━
```

### 技术挑战

| 挑战 | 说明 | 影响程度 |
|------|------|---------|
| **状态竞争** | 多个 runner 同时读写 `tasks.json` | 🔴 高 |
| **文件冲突** | 多个 generator 可能修改同一文件 | 🔴 高 |
| **API 限制** | 并行调用会更快触发 rate limit | 🟡 中 |
| **Token 预算** | 需要原子性的计数器 | 🟡 中 |
| **日志交错** | 多任务日志混合，难以追踪 | 🟢 低 |
| **错误传播** | 任务失败需要通知依赖它的任务 | 🟡 中 |

### 实现方案

**方案一：层级批量并行**（推荐起步方案）

```typescript
async getReadyTasks(maxParallel: number): Promise<Task[]> {
  // 返回所有 dependencies 已满足的 pending 任务
  // 限制最大并行数
}
```

执行逻辑：
1. 获取所有就绪任务（依赖已满足）
2. 用 `Promise.all` 并行执行
3. 等待整批完成后，再获取下一批

**方案二：动态调度器**

使用队列 + worker 模式，更灵活但实现更复杂。

### 文件冲突解决方案

这是最大的技术挑战：

1. **静态分析依赖**：Planner 阶段就标注每个任务会修改的文件列表，运行时检测冲突
2. **乐观锁 + 重试**：允许并行，冲突时检测并重新执行
3. **文件级锁**：任务执行前获取要修改文件的锁

### 分阶段实施

1. **Phase 1**：实现层级并行，同一层级任务并行执行
2. **Phase 2**：引入文件锁机制解决文件冲突
3. **Phase 3**：动态调度，最大化并行度

## 优化方向二：流水线并行

### 核心思想

有依赖的任务之间也可以实现一定程度的并行——通过"部分完成通知"机制。

```
传统模式（等待完全完成）：
Task A: [Generator ──────┐ Evaluator ───┐]
                                  ↓
Task B:                          [Generator ────── Evaluator]

流水线模式（提前衔接）：
Task A: [Generator ─────┼──→ mailbox ──→] [Evaluator]
                              ↓
Task B:                 [Generator ←──── 接收接口定义]
```

**关键洞察**：后端不需要等前端完全验收通过，只要前端产出了接口契约，后端就可以开始。

### 参考实现：Claude Agent SDK Agent Teams

Claude Agent SDK 提供了 Agent Teams 概念：
- 启动多个 agent
- agent 之间通过 Mailbox 互相通信
- 实现协作式的任务执行

### 通信内容

mailbox 可能传递的内容：

1. **接口契约**（前后端对接）
   - API endpoint 定义
   - 数据结构 / 类型定义

2. **状态通知**
   - "我已经完成了 X 模块"
   - "我遇到一个问题，需要讨论"

3. **约束同步**
   - "数据库 schema 已定，请遵循"
   - "这个函数签名已导出，你可以调用"

## 优化方向三：Agent Teams 单任务内协作

### 概念

即使单个任务，也可以用 Agent Teams 加速执行：

```
传统单 Agent：
[Generator] 做所有事情：写代码、写测试、写文档...

Agent Teams：
[前端Agent] ←→ [后端Agent] ←→ [测试Agent]
      ↑              ↓              ↑
      └────── mailbox 通信 ─────────┘
```

### 价值对比

| 维度 | 传统串行 | Agent Teams |
|------|----------|-------------|
| **协作模式** | 隔离执行 | 持续协调 |
| **信息流动** | 批量传递 | 实时同步 |
| **错误发现** | 延迟到评估阶段 | 早期发现 |
| **人机类比** | 流水线工人 | 真实团队讨论 |

## 技术挑战

### 1. 部分完成的定义

任务何时算"足够完成"可以通知下游？

```typescript
interface TaskProgress {
  interfacesDefined: boolean;  // API 契约已定义
  coreLogicComplete: boolean;  // 核心逻辑完成
  testsPassing: boolean;       // 测试通过
  reviewComplete: boolean;     // 验收完成
}
```

### 2. 回滚与错误传播

**场景**：
```
前端发了接口契约 → 后端开始实现 → 前端评估失败需要改接口
```

**问题**：后端已经写了一半的代码怎么办？

**方案**：需要版本化的契约 + 影响范围评估

### 3. SDK 能力调研

需要确认 `@anthropic-ai/claude-agent-sdk` 是否支持 agent 间通信，或者需要自行实现。

## 实施路线图

```
Level 1: 任务级并行
         └── 兄弟节点并行执行
         └── 相对简单，可快速实现

Level 2: 流水线并行
         └── 有依赖的任务可以提前开始
         └── 需要部分完成通知机制
         └── 中等复杂度

Level 3: Agent Teams 协作
         └── 多 agent 实时通信
         └── 单任务内也能并行
         └── 需要深入 SDK 能力
```
