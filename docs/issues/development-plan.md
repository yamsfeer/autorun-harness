## 第一批：立即做（1-2 天，投入小收益大）

> 这批改动小、风险低、效果明确，可以快速落地。

### 1.1 max-tasks 默认值改为全部执行

| 字段 | 内容 |
|------|------|
| **对应 issue** | S-4 |
| **改动** | `src/commands/run.ts:41` 和 `src/core/orchestrator.ts:321` 默认值从 `10` 改为 `Infinity` |
| **预期效果** | 用户不再需要每次手动指定 `--max-tasks` |
| **风险** | 极低，仅改默认值，用户仍可通过参数覆盖 |

### 1.2 Git tag 做回滚锚点

| 字段 | 内容 |
|------|------|
| **对应 issue** | R-1 |
| **改动** | `src/core/orchestrator.ts`：任务开始前 `git tag pre-task-${id}`，完成后 `git add -A && git commit -m "task-${id}: ${title}"` |
| **预期效果** | 出错时可精确定位变更范围、可回退到任务前状态 |
| **风险** | 低，git 操作不影响业务逻辑。需处理 git 不存在的边界情况 |

### 1.3 Git diff 注入上下文

| 字段 | 内容 |
|------|------|
| **对应 issue** | R-2 |
| **改动** | `src/core/orchestrator.ts`：构建 Generator prompt 前执行 `git log --oneline -5` 和 `git diff HEAD~1 --stat`，将结果注入 |
| **预期效果** | Generator 能看到最近改了什么，减少重复劳动和与已有代码冲突 |
| **风险** | 低，仅增加只读信息注入。需处理首次运行无 git 历史的情况 |

### 1.4 emoji 图标优化

| 字段 | 内容 |
|------|------|
| **对应 issue** | S-2 |
| **改动** | `src/core/message-handler.ts`：将 emoji 替换为 ASCII 符号或文字标识（如 `[READ]` `[WRITE]` `[EDIT]` `[BASH]`），或检测终端能力后选择方案 |
| **预期效果** | 消除终端乱码问题 |
| **风险** | 极低，仅影响日志展示 |

### 1.5 init + run 连续执行

| 字段 | 内容 |
|------|------|
| **对应 issue** | S-3 |
| **改动** | 添加 `autorun-harness start`（或 `autorun`）命令，内部依次调用 init 和 run |
| **预期效果** | 用户一键完成从初始化到执行的完整流程 |
| **风险** | 极低，组合已有逻辑 |

## 第二批：核心质量提升（3-5 天，解决最痛的问题）

> 这批解决三个实际项目运行中暴露的核心痛点：上下文断裂、评估不可靠、中断后无法恢复。

### 2.1 依赖任务文件列表注入上下文

| 字段 | 内容 |
|------|------|
| **对应 issue** | Q-1 |
| **改动** | (1) `src/types/index.ts`：Task 增加 `produced_files` 和 `key_decisions` 字段 (2) `src/core/orchestrator.ts`：Generator prompt 注入上游任务的文件列表和关键决策 (3) `prompts/generator.md`：说明如何使用注入的上下文 (4) `prompts/evaluator.md`：完成时提取 produced_files/key_decisions |
| **预期效果** | Generator 知道上游写了什么代码、做了什么决策，减少因上下文缺失导致的错误。这是三个项目 bug report 中大多数问题的根源 |
| **风险** | 低-中。需注意上下文长度控制（文件列表可能很长），可限制为只注入直接依赖（非传递依赖） |

### 2.2 "未找到评估报告" 警告修复

| 字段 | 内容 |
|------|------|
| **对应 issue** | H-2 |
| **改动** | `src/core/evaluator.ts:150`：调查为什么评估器成功完成但未写入报告文件。报告缺失时区分"评估器故障"和"代码不合格"，不应默认生成失败报告 |
| **预期效果** | 评估结果更准确，减少因评估器自身问题导致的误判 |
| **风险** | 低，修改的是边界情况的错误处理逻辑 |

### 2.3 代码存在性校验 + --force-continue

| 字段 | 内容 |
|------|------|
| **对应 issue** | R-3, R-4 |
| **改动** | (1) `src/core/state-manager.ts`：依赖检查时，如 `status !== "completed"`，额外检查 `outputs` 文件是否存在 (2) 新增 `--fix-status` 命令：扫描文件系统与 tasks.json 对比，自动修正状态 (3) 新增 `--force-continue` 选项：忽略 needs_human 状态继续执行下游任务 |
| **预期效果** | 解决中断重启后的大规模阻塞问题（三个项目共 28 个任务被不必要阻塞） |
| **风险** | 中。需要仔细设计"代码已存在"的判断标准（文件存在 ≠ 功能正确），建议保守策略 + 用户可配置 |

### 2.4 强制质量门（tsc + ESLint）

| 字段 | 内容 |
|------|------|
| **对应 issue** | Q-5 |
| **改动** | `src/core/orchestrator.ts`：Generator 完成后自动运行 `tsc --noEmit` + `eslint`，任一失败即 block 并反馈给 Generator 修复。支持项目级配置（有 tsconfig 才跑 tsc，有 .eslintrc 才跑 eslint） |
| **预期效果** | 确定性检查优先于 AI 判断，类型错误和风格问题不会流入后续任务 |
| **风险** | 中。需要处理不同项目技术栈的差异，建议先支持 TypeScript + ESLint，后续扩展 |

### 2.5 人工介入接口

| 字段 | 内容 |
|------|------|
| **对应 issue** | H-1 |
| **改动** | (1) 任务标记 needs_human 时暂停执行，输出通知信息（当前先做终端提示 + 日志记录，邮件/Webhook 后续扩展） (2) 新增 `autorun-harness resume` 命令：人工修复后恢复自动执行 (3) 新增 `autorun-harness mark <taskId> completed` 命令：手动标记任务完成并解锁下游 |
| **预期效果** | needs_human 不再是项目死胡同，有明确的恢复路径 |
| **风险** | 低。核心是命令行工具 + 状态管理，不涉及复杂集成 |

## 第三批：质变型改进（1-2 周，需要较多设计）

> 这批改动范围较大，需要一定的设计和验证。但完成后将带来质的提升。

### 3.1 周期性全量回归测试

| **对应 issue** | Q-3 |
| **改动** | `src/core/orchestrator.ts`：主循环中增加计数器，每完成 N 个任务（默认 3，可配置）执行 `npm run build && npm test`。回归失败时暂停并反馈修复 |
| **前置条件** | 2.4（质量门）已完成，可复用其构建/测试执行框架 |
| **预期效果** | 错误不再积累到后期才被发现 |
| **风险** | 中。需要处理"项目没有测试"的情况，回归失败时的修复策略需要设计 |

### 3.2 验收标准变成可执行测试用例

| 字段 | 内容 |
|------|------|
| **对应 issue** | Q-2 |
| **改动** | (1) `prompts/planner-full.md`：要求 Planner 同时生成测试用例文件 (2) `src/core/evaluator.ts`：核心逻辑从"AI 主观评估"改为"运行测试 + 分析结果" (3) `src/types/index.ts`：AcceptanceCriterion 增加测试文件路径字段 |
| **前置条件** | 3.1（回归测试）的测试运行框架可复用 |
| **预期效果** | 评估可信度质变，消除 AI 评估 AI 的主观偏差 |

## 验证策略

每批完成后，用真实项目验证：

1. **第一批完成后**：运行一个 10+ 任务的项目，验证 git tag/commit、diff 注入是否正常工作
2. **第二批完成后**：重新运行之前失败的项目（维修派单管家 / 洁邻管家 / 万家官网管家），对比改进前后的通过率和阻塞情况
3. **第三批完成后**：选择一个新项目从零到一跑完整流程，验证质量门 + 回归测试 + 可执行测试用例的配合效果
