### Q-1: 依赖任务文件列表注入上下文

| 字段 | 内容 |
|------|------|
| **来源** | issues-4-improvement-plan.md（改进 1） |
| **提出原因** | 分析发现：每次 Generator 是独立会话，只拿到当前任务 JSON + spec.md，不知道上游任务写了什么代码、做了什么决策。越往后的任务，上下文缺失越严重，出错概率指数级上升 |
| **状态** | 未实施 |
| **优先级** | P0 |
| **难度** | 简单 |

**具体内容**：

`tasks.json` 中已有 `dependencies` 字段，但 Generator 未使用。每个任务完成后应记录 `produced_files`（产生/修改的文件列表）和 `key_decisions`（关键决策），Generator 启动时根据当前任务的依赖关系，自动注入上游任务的上下文。

**改动范围**：
- `src/types/index.ts`：Task 类型增加 `produced_files` 和 `key_decisions` 字段
- `src/core/orchestrator.ts`：Generator prompt 构建逻辑，注入依赖任务的上下文
- `prompts/generator.md`：提示词中说明如何使用注入的上下文
- `prompts/evaluator.md`：评估器在任务完成时提取 `produced_files` 和 `key_decisions`

**备注**：代码中已有 `Task.outputs` 字段，但仅用于 Bug-006 的依赖满足检查（检查文件是否存在），未用于 Generator prompt 注入。

## 二、错误恢复与可回滚性

### R-1: Git tag 做回滚锚点

| 字段 | 内容 |
|------|------|
| **来源** | issues-4-improvement-plan.md（改进 6） |
| **提出原因** | 框架没有回滚能力。任务出错后只能重试同一上下文，无法回到出错前的状态。Bug Report 中多次出现"代码已存在但任务标记为失败"的情况，如果有回滚锚点可以大幅降低修复成本 |
| **状态** | 未实施 |
| **优先级** | P1 |
| **难度** | 很简单 |

**具体内容**：

每个任务开始前 `git tag "pre-task-${taskId}"`，完成后 `git add -A && git commit`。出错时可 `git diff pre-task-3..HEAD` 精确定位变更范围，或 `git reset --soft pre-task-3` 回退。

**改动范围**：
- `src/core/orchestrator.ts`：任务开始前打 tag，完成后 commit

## 四、并行执行优化

### P-1: 任务级并行（兄弟节点并行执行）

| 字段 | 内容 |
|------|------|
| **来源** | issue-improvement-parallel-execution.md |
| **提出原因** | 当前顺序执行模式未充分利用任务间并行潜力。不依赖其他任务的任务（兄弟节点）之间可以并行执行，能显著缩短总体执行时间 |
| **状态** | 未实施 |
| **优先级** | P2 |
| **难度** | 中等 |

**具体内容**：

推荐起步方案——层级批量并行：
1. 获取所有就绪任务（依赖已满足）
2. 用 `Promise.all` 并行执行
3. 等待整批完成后，再获取下一批

技术挑战：状态竞争（多 runner 读写 tasks.json）、文件冲突（多 generator 修改同一文件）、API 限制、Token 预算。

### P-4: 文件冲突检测机制

| 字段 | 内容 |
|------|------|
| **来源** | issue-improvement-parallel-execution.md |
| **提出原因** | 并行执行时多个 generator 可能修改同一文件，这是最大的技术挑战 |
| **状态** | 未实施 |
| **优先级** | P2（P-1 的前置条件） |
| **难度** | 中等 |

**具体内容**：

三种方案：
1. 静态分析依赖：Planner 阶段标注每个任务会修改的文件列表，运行时检测冲突
2. 乐观锁 + 重试：允许并行，冲突时检测并重新执行
3. 文件级锁：任务执行前获取要修改文件的锁
