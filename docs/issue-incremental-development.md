# Issue: 现有项目的增量开发支持不足

**Created**: 2026-04-16
**Status**: Open (待讨论与规划)
**Priority**: High

---

## 问题描述

当前系统的设计核心假设是"从零到一"（greenfield）开发：
- Planner 接收 PRD，生成 `spec.md` 和 `tasks.json`
- Generator 基于空白/接近空白的项目目录进行实现
- 每个 task 默认是"创建新文件/新模块"的建构性任务

然而，实际开发场景中大量需求发生在**已有项目**上，包括：
1. **新增功能**：在现有系统中添加新模块
2. **修改功能**：调整已有功能的实现逻辑
3. **重构**：跨文件的重构和架构调整

在这些增量场景下，当前系统暴露出根本性不适应。

---

## 根本差距分析

### 缺少"代码库理解"阶段

当前流程：
```
PRD → Planner → spec + tasks → Generator → Evaluator
```

增量场景需要的流程：
```
需求描述 → 【代码库分析】→ 影响范围分析 → Planner → spec + tasks → Generator → 【回归测试】→ Evaluator
```

### 核心缺口

| 缺口 | 说明 |
|------|------|
| **前置分析缺失** | 没有机制让系统先理解现有架构、代码模式、依赖关系 |
| **任务模型单一** | `tasks.json` 只支持"创建型"任务，没有 `modify` / `refactor` / `delete` 类型 |
| **上下文不精确** | Generator 无法获得与当前修改相关的已有代码上下文 |
| **后置回归缺失** | 修改后没有机制验证是否破坏了已有功能 |

---

## 三种增量场景的复杂度

| 场景 | 核心难点 | 当前支持度 |
|------|---------|-----------|
| **新增功能** | 需要理解现有架构、融入现有模式、不破坏已有功能 | 弱 — 缺少代码库分析 |
| **修改功能** | 需要精确定位影响范围、理解修改的连锁反应 | 无 — 没有 impact analysis |
| **重构** | 需要全局视角、渐进式修改、每步都要回归验证 | 无 — 没有 diff/regression 机制 |

---

## 待讨论的可能思路

### 思路 1：增加"代码库理解"阶段

在 Planner 之前增加一个 Explorer Agent：
```
现有项目 + 需求 → Explorer Agent → 代码库概览 → Planner → Generator → Evaluator
```

**风险**：大代码库的全量分析会超出 context window，需要结构化摘要。

### 思路 2：需求驱动的定向分析

从需求出发，只查找和分析相关代码：
```
需求 → 定向查找关键文件 → 定向读取代码 → Planner → Generator
```

**风险**：可能漏掉间接依赖，导致修改引发连锁故障。

### 思路 3：分两轮规划（推荐重点讨论）

- **第一轮（轻量探索）**：快速扫描项目结构 + 依赖关系，产出"影响范围评估"
- **第二轮（深度规划）**：基于影响范围定向读取代码，生成精确的增量 tasks

---

## 关键开放问题

这些问题需要在进入方案设计前讨论清楚：

1. **任务模型扩展**：如何定义 `create` / `modify` / `refactor` / `delete` 四种任务类型，以及它们各自的上下文和验证方式？
2. **上下文预算控制**：已有项目可能包含几百个文件，如何决定 Generator 应该看到哪些代码？
3. **回归验证机制**：如果现有项目测试覆盖不足，如何确认修改没有破坏已有功能？
4. **增量粒度适配**：系统如何判断一个需求的复杂度，并自动选择轻量还是深度策略？
5. **与现有改进计划的衔接**：这与 improvement-plan.md 中提到的"上下文断裂"、"评估器可信度"、"错误积累"问题高度相关，如何统筹解决？
